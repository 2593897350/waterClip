# 图片去水印网站 MVP 设计文档

## 1. 目标

构建一个面向照片与社交媒体图片的图片去水印网站 `MVP`。首版目标是让用户无需登录即可上传图片，系统先自动检测疑似水印区域，用户可手动修正选区，然后选择 `极速模式` 或 `高清模式` 执行去水印并下载结果。

本版本的核心不是做全功能平台，而是验证以下三件事：

1. 用户是否愿意为“自动检测 + 手动修正”的去水印流程买单。
2. 面向照片类复杂背景时，双模式输出是否能兼顾速度与质量。
3. `Go API + Python 图像处理引擎` 的服务拆分是否足够稳定、易于后续迭代。

## 2. 范围

### 2.1 In Scope

- 首页上传图片
- 自动检测疑似水印区域
- 编辑页展示检测蒙版
- 用户使用画笔/橡皮擦修正蒙版
- 选择 `极速模式` 或 `高清模式`
- 发起去水印处理任务
- 展示原图与结果图对比
- 下载处理结果
- 失败后重试或返回继续修正

### 2.2 Out of Scope

- 登录、注册、个人中心
- 支付、会员、额度体系
- 批量上传与批量处理
- 历史记录与长期图片保存
- 复杂后台管理
- 对所有图片场景的效果承诺

## 3. 产品结构

首版采用 `工具型 MVP` 结构，页面最少但路径完整：

1. `首页`
作用：承接访问流量、说明能力、提供上传入口。

2. `编辑页`
作用：展示自动检测结果，允许用户修正蒙版并选择处理模式。

3. `结果页`
作用：展示处理结果、支持下载、允许返回编辑页继续修正。

整体路径固定为：

`首页上传 -> 自动检测 -> 编辑修正 -> 选择模式 -> 去水印处理 -> 结果下载/继续修正`

## 4. 核心交互

### 4.1 首页

首页保持简洁，以工具转化为优先：

- 主标题直接表达“图片去水印”
- 强调 `自动检测 + 手动修正`
- 强调 `极速 / 高清` 双模式
- 主按钮为上传图片
- 可补充少量前后对比示例和常见问题

首页不承载复杂营销内容，不做长篇介绍，不引入注册转化流程。

### 4.2 编辑页

编辑页是首版核心页面，必须完整支撑去水印决策。

编辑页包含：

- 图片主画布
- 自动检测出的疑似水印蒙版
- 画笔工具：补画遗漏区域
- 橡皮擦工具：去掉误判区域
- 缩放与重置
- `极速模式` / `高清模式` 选择
- 开始处理按钮

设计原则：

- 自动检测先给建议，但不强制用户接受
- 用户修正成本要低，避免复杂专业参数
- 用户处理失败后应保留当前蒙版，不重新开始

### 4.3 结果页

结果页包含：

- 结果图展示
- 原图/结果图对比
- 下载按钮
- 返回编辑页继续修正

如果处理结果不满意，返回编辑页时保留原图与上次蒙版，避免重新上传。

## 5. 模式定义

### 5.1 极速模式

目标：在几秒内给出结果，优先降低等待成本。

适用场景：

- 用户只想快速试一下
- 图片背景相对简单
- 用户愿意接受少量修复瑕疵

预期特征：

- 处理链路较轻
- 返回更快
- 效果上限低于高清模式

### 5.2 高清模式

目标：尽量得到更自然的结果，优先保证修复质量。

适用场景：

- 社交媒体照片
- 复杂背景或纹理区域
- 用户希望下载可用成品

预期特征：

- 处理链路更重
- 等待时间更长
- 结果质量优先于速度

## 6. 技术架构

首版采用以下模块拆分：

### 6.1 Next.js 前端

职责：

- 上传图片
- 展示自动检测返回的蒙版
- 提供蒙版编辑能力
- 切换 `极速` / `高清`
- 提交处理请求
- 轮询或查询任务状态
- 展示结果并下载

边界：

- 不负责真正的去水印算法
- 只做轻量交互与画布层处理

### 6.2 Go API 服务

职责：

- 暴露 HTTP 接口
- 接收图片上传
- 创建检测任务与去水印任务
- 管理任务状态
- 保存并组织文件路径
- 调用 Python 图像处理服务
- 返回结果地址与错误信息

边界：

- 不实现核心图像算法
- 是前端与处理引擎之间的编排层

### 6.3 Python 图像处理引擎

职责：

- 自动检测疑似水印区域
- 输出标准化蒙版
- 基于最终蒙版执行去水印修复
- 区分 `极速链路` 与 `高清链路`

边界：

- 专注图像处理与模型/算法试错
- 不承载业务路由、鉴权、页面逻辑

### 6.4 文件存储

首版保存以下文件：

- 原图
- 自动检测蒙版
- 用户修正版蒙版
- 结果图

首版采用本地存储即可，并配套定时清理机制。后续再迁移到对象存储。

## 7. 任务流

完整任务流分为两段。

### 7.1 检测任务

1. 用户在首页上传图片
2. 前端调用 `Go API` 创建检测任务
3. `Go API` 保存原图并调用 `Python` 执行自动检测
4. `Python` 返回初始蒙版
5. `Go API` 将检测结果返回前端
6. 前端进入编辑页并展示蒙版

### 7.2 去水印任务

1. 用户在编辑页修正蒙版
2. 用户选择 `极速模式` 或 `高清模式`
3. 前端提交原图标识、最终蒙版与模式到 `Go API`
4. `Go API` 创建去水印任务并调用 `Python`
5. `Python` 输出结果图
6. `Go API` 返回任务完成状态与结果地址
7. 前端展示结果页

## 8. 接口与数据边界

首版不需要先写完整 OpenAPI，但需要统一 4 类核心对象：

1. `Image Asset`
表示原图与结果图等文件资源。

2. `Mask`
表示自动检测或用户修正后的水印区域数据。

3. `Task`
表示检测任务或去水印任务的状态。

4. `Mode`
枚举值：`fast`、`hd`

首版至少需要这些接口能力：

- 上传图片并创建检测任务
- 获取检测结果
- 提交最终蒙版并创建去水印任务
- 查询去水印任务状态
- 获取结果图地址

## 9. 错误处理

首版只处理最关键、用户最能感知的失败分支。

### 9.1 上传失败

场景：

- 图片格式不支持
- 文件过大
- 文件损坏

处理：

- 明确提示原因
- 允许重新上传

### 9.2 自动检测不准确

场景：

- 未检测到明显水印
- 检测区域过大或过小

处理：

- 允许用户手动补画或擦除
- 不阻断后续去水印流程

### 9.3 去水印失败

场景：

- Python 处理异常
- 任务超时
- 结果文件生成失败

处理：

- 保留当前蒙版
- 允许用户重试
- 允许切换 `极速` / `高清` 再试

### 9.4 结果不满意

场景：

- 去水印区域仍有残留
- 背景修复出现不自然痕迹

处理：

- 返回编辑页继续修正
- 不要求重新上传

## 10. 非功能要求

### 10.1 性能

- `极速模式` 应尽量在短时间内返回结果
- `高清模式` 允许明显更长的处理时间，但要给出处理中反馈
- 前端不能在处理期间失去状态

### 10.2 可扩展性

- Go API 与 Python 引擎通过清晰接口通信，便于后续替换算法
- 文件存储路径设计保持可迁移，后续可替换为对象存储
- 任务模型保留扩展到队列化执行的空间

### 10.3 可维护性

- 前端、API、图像处理服务职责清晰
- 不把算法细节塞进 Go API
- 不把业务状态管理塞进 Python 服务

## 11. 测试策略

首版测试只覆盖主价值链路。

### 11.1 前端测试

- 上传流程可用
- 蒙版展示可用
- 画笔/橡皮擦修正可用
- 模式切换可用
- 结果展示与下载可用

### 11.2 Go API 测试

- 任务创建成功
- 任务状态流转正确
- 文件路径与结果关联正确
- 错误场景返回明确

### 11.3 Python 引擎测试

- 自动检测输出格式正确
- 去水印输出结果文件正确生成
- `fast` / `hd` 两种模式均能执行

### 11.4 端到端测试

至少准备一张真实照片，验证以下链路完整跑通：

`上传 -> 自动检测 -> 手动修正 -> 发起处理 -> 查看结果 -> 下载`

## 12. 后续演进方向

如果 `MVP` 验证有效，后续优先级建议如下：

1. 历史记录与临时任务恢复
2. 对象存储与更稳定的结果访问
3. 登录与额度控制
4. 批量处理
5. 支付与会员体系
6. 后台管理与内容审核

## 13. 结论

首版应坚持“小而完整”的工具型方案：

- 页面只保留首页、编辑页、结果页
- 用户无需登录
- 自动检测提供建议，手动修正负责兜底
- `Go API` 负责编排
- `Python` 负责图像处理
- 重点验证用户是否认可这条去水印链路，而不是过早建设平台能力
